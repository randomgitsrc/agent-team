## 经验总结

### Discord Agent 绑定机制
- **Bot Token**：1个，全系统共用
- **Guild (服务器)**：Discord 服务器，有 ID
- **Channel (频道)**：服务器内的子频道，有 ID
- **Bindings**：匹配规则 `{agentId, match: {channel, accountId}}`
- **accountId 映射**：
  - DM → `"dm"`
  - 频道 → 频道 ID
  - 服务器 → 服务器 ID

### 多 Bot vs 单 Bot
| 模式 | 单 Bot | 多 Bot |
|------|--------|--------|
| 频道单聊 | ✅ 每频道绑定不同 agent | ✅ |
| 私信 | ✅ 绑定 main | ✅ 各 bot 独立 |
| 群聊多 Agent | ❌ 只能 @同一个 bot | ✅ @bot1 / @bot2 |

### mentionPatterns 方案
- 位置：`agents.list[].groupChat.mentionPatterns`
- 风险：误触发、维护成本、体验不稳定
- 结论：暂不采用

### Discord vs Telegram
| 平台 | 难度 | 推荐场景 |
|------|------|----------|
| Telegram | ⭐⭐⭐⭐⭐ | 个人 AI 助手 |
| Discord | ⭐⭐⭐⭐ | 社区、多 Agent、并行任务 |

### 频道自动创建/销毁
- 支持创建线程/频道（CLI）
- 不支持自动根据任务创建，需写 Hook/Cron

## 新经验：跨 Agent 调度

### 能力验证
- 阿九（main）通过 sessions_send 调度司命（commander）主动在 Discord 频道发消息
- 司命调用 message 工具成功发送到频道

### 配置要点
```json
{
  "tools.sessions.visibility": "all",
  "tools.agentToAgent.enabled": true
}
```

### 关键教训
- 给司命的指令要**拆成明确的三步**：
  1. 第一步：生成 YAML 内容
  2. 第二步：调用 message 工具发送到频道（附上具体参数）
  3. 第三步：回复"已发送"
- 如果只说"发送到频道"，司命会误解为"输出文字给用户看"，而不是真正调用工具

### 经验总结

#### 后台任务管理
- 启动后台任务 → 在 TaskFlow 创建任务（ID=XX）
- 状态=进行中 → 完成后更新为已完成 + 写总结日志
- **强制检查点**：标记完成前必须确认：进度=100%、有总结日志、实测验证通过

#### 测试验收规范（新增到 AGENTS.md）
- subagent 说通过 ≠ 真的通过，必须自己实测
- UI/交互：必须打开浏览器实际点击测试
- API：必须 curl 或代码验证

#### 技术选型原则
- 选熟悉的：不熟悉的技术栈容易踩坑
- 复杂组件：优先用简单方案替代（如 NDropdown → NSelect）
- 及时看日志：浏览器 console / 终端输出

#### 任务复杂度管理
- 任务 <5 分钟：直接完成
- 任务 5-15 分钟：设 timeoutSeconds=600
- 任务 >15 分钟：拆成多个子任务

### 教训

2. 状态按钮 Bug：subagent 说通过但实际没生效 → 后来重新修复
3. 任何代码改动后必须自己实测验证
