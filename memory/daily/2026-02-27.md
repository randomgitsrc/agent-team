## 📌 后续更新 (20:23) - Claude Code 操控经验

### 重大进展：成功操控 Claude Code ✅

**工作流验证**: "我设计需求 → Claude Code 实现 → 我审查结果" 模式完全可行

#### 核心命令
```bash
# 基本调用（自动执行，无需交互确认）
claude --dangerously-skip-permissions "你的任务"

# 在指定目录执行
cd ~/project/xxx && claude --dangerously-skip-permissions "任务"

# 长任务（后台执行）
bash pty:true workdir:~/project background:true timeout:300 command:"claude --dangerously-skip-permissions '任务'"
```

#### 关键参数说明
| 参数 | 作用 | 场景 |
|------|------|------|
| `--dangerously-skip-permissions` | 跳过所有权限确认 | 自动化执行必需 |
| `pty:true` | 伪终端模式 | Claude Code 需要 TTY 时 |
| `background:true` | 后台运行 | 长任务不阻塞 |
| `timeout:300` | 5分钟超时 | 防止无限等待 |

#### 环境配置
- `ANTHROPIC_API_KEY` - API key（通过 `.env` 设置，OpenClaw 自动加载）
- `ANTHROPIC_BASE_URL` - 可选，指向代理服务（如 Moonshot）
- 重启后生效：`openclaw gateway restart`

#### 最佳实践
1. **任务粒度**: 一次一个明确任务，复杂任务拆解
2. **自测能力**: Claude Code 可用 playwright/puppeteer 自测截图
3. **文件位置**: 在 `~/project/ccworkspace/` 下创建项目目录
4. **迭代优化**: 不满意继续让 Claude Code 修改，直到满足

#### 成功案例
- ✅ lucky-button 项目：创建 HTML 页面 + 键盘快捷键优化
- ✅ 两种命运效果：新世界 vs 末日的视觉差异实现
- ✅ 无需手动编写代码，纯 Claude Code 实现

#### 不适用场景
- 简单一行命令能完成的事（直接自己做更快）
- `~/clawd` 工作区（SKILL.md 规定禁止）

## 📌 后续更新 (21:40) - 迷宫游戏开发全流程经验

### 项目概况
- **任务**: 可爱迷宫游戏（单文件 HTML）
- **方式**: 阿九调度 Claude Code 后台开发
- **耗时**: 从启动到最终验收约 50 分钟（含踩坑）
- **迭代**: 3 轮（初版 → 优化 → bug 修复 × 2）

---

### 一、规则（写入 AGENTS.md 的硬规则）

#### R1: 后台任务必须走 wrapper
```bash
# 正确 ✅
~/.openclaw/workspace/scripts/bg_task.sh "任务描述" claude --dangerously-skip-permissions '...'

# 错误 ❌
claude --dangerously-skip-permissions '...'  # 裸起，无通知保障
```
**原因**: 不走 wrapper → 进程超时/失败 → 老板等 5+ 分钟无人告知

#### R2: 不手改 Claude Code 的代码
阿九的职责是**调度和验收**，不是当码农：
- ❌ 自己 sed/edit 改代码 → 改着改着越来越乱，还引入新 bug
- ✅ 写清楚 bug 描述 → 让 Claude Code 修 → 验收结果

#### R3: 收到系统事件必须响应
收到 `[System Message] Exec completed/failed` 后：
- 必须检查结果
- 必须给老板发消息
- 禁止 NO_REPLY

---

### 二、决策记录

| 决策 | 选择 | 原因 |
|------|------|------|
| 通知机制 | bg_task.sh wrapper + prompt 内双保险 | shell 层确定性 > AI 自觉性 |
| 软规则 vs 硬机制 | 硬机制优先 | "叫 AI 记得做某事" 不可靠，写进 AGENTS.md 是文字，写进脚本是代码 |

| Claude Code 交互确认 | 需改进 | PTY 菜单选择不稳定，数字键选择也不靠谱，两次选错 "No, exit" |

---

### 三、工作流（验证后的标准流程）

```
1. 需求分析
   └─ 拆解为 Claude Code 能一次完成的粒度

2. 启动任务
   └─ bg_task.sh wrapper + pty:true + timeout + workdir
   └─ prompt 末尾加 openclaw system event（双保险 A）

3. 等待 + 监控

4. 收到完成通知
   └─ 验收代码逻辑（grep 关键函数、检查结构）
   └─ 不手改，有问题写新 prompt 让 Claude Code 修

5. 交付
   └─ message send 告知老板结果
   └─ 附截图（如果有）
```

### 四、踩坑记录

#### 坑1: PTY 菜单交互不稳定
**现象**: Claude Code 启动时有两个确认菜单（trust folder + bypass permissions），send-keys Down+Enter 两次都选成了 "No, exit"
**原因**: PTY 终端菜单的焦点状态不确定，Down 键可能没生效
**临时方案**: 多试几次，或用数字键
**待优化**: 研究 `--dangerously-skip-permissions` 是否能跳过所有交互

#### 坑2: 手改代码引入新问题
**现象**: 猫咪定位 bug，我手动 sed 加了 `position: relative`，搞出重复属性，还没解决根本问题
**教训**: Claude Code 看全局比我手改精准，且不会遗漏关联修改

#### 坑3: renderMaze() 清空 innerHTML 把 player 也删了
**现象**: 上一轮把 #player 移入 #maze 修了定位，但 renderMaze() 的 innerHTML='' 把它也清了
**教训**: 修一个 bug 可能引入另一个，需要完整上下文理解

#### 坑4: 通知延迟 = 通知缺失
**现象**: Claude Code 20:52 写完文件，20:54:55 超时退出，到 20:57 老板主动问才知道
**根因**: 无兜底通知 → 修复方案: bg_task.sh wrapper

### 五、Claude Code Prompt 工程经验

#### 好的 prompt 结构
```
1. 明确任务（Fix / Create / Iterate）
2. 具体问题描述（现象 + 原因分析）
3. 建议的修复方向（但不限制）
4. 约束条件（不要破坏 XXX）
5. 完成后执行通知命令
```

#### 有效技巧
- 让 Claude Code 先读文件再改（"Read the FULL file first"）
- 让它自己用 headless Chrome 截图验证
- 给中文翻译时提供完整对照表，不要只说"改成中文"
- Bug 描述时给出**现象 + 代码位置 + 建议方向**，比只说"不工作了"有效 10 倍

### 六、bg_task.sh 技术细节

**位置**: `~/.openclaw/workspace/scripts/bg_task.sh`
**机制**: 
1. 接收描述 + 命令
2. 执行命令，捕获退出码
3. 计算耗时
4. 区分成功(0) / 超时(143) / 失败(其他) 
5. `openclaw system event --mode now` 即时通知
**延迟**: 5-15 秒（event → OpenClaw 唤醒 → 生成回复 → Telegram 送达）
